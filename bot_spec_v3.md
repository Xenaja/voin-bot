## 🗄️ ХРАНИЛИЩЕ СОСТОЯНИЙ (store.js)

Использовать **SQLite** (библиотека `better-sqlite3`).

### Таблица `users`

| Поле | Тип | Описание |
|---|---|---|
| `id` | INTEGER PK | авто |
| `platform` | TEXT | `'telegram'` или `'vk'` |
| `chat_id` | TEXT | уникальный ID пользователя на платформе |
| `state` | TEXT | текущее состояние (см. ниже) |
| `updated_at` | DATETIME | время последнего изменения состояния |
| `reminder_count` | INTEGER | `0`, `1`, `2` — сколько напоминаний отправлено (24ч и 48ч) |

### Состояния пользователя

| Состояние | Когда устанавливается |
|---|---|
| `MSG1_SENT` | Отправлено MSG_1 (пользователь запустил бота) |
| `MSG2_SENT` | Отправлено MSG_2 (нажал кнопку или авто-продвижение через 30 мин) |
| `AWAIT_PAYMENT` | Отправлено MSG_3 (нажал кнопку или авто-продвижение через 30 мин) |
| `COMPLETED` | Пользователь написал ГОТОВО и получил файлы |

### Методы store.js

```js
getUser(platform, chatId)            // → объект пользователя или null
upsertUser(platform, chatId, state)  // создать или обновить запись + updated_at
getPendingAutoProgress(platform)     // → пользователи, у которых updated_at < now - 30 мин
                                     //   и state IN ('MSG1_SENT', 'MSG2_SENT')
getPendingReminders(platform)        // → пользователи для 24-часового напоминания
                                     //   state = 'AWAIT_PAYMENT', reminder_sent = 0
incrementReminderCount(platform, chatId) // увеличить reminder_count на 1
```

### SQL-запросы для планировщика

```sql
-- Авто-продвижение (30 минут)
SELECT * FROM users
WHERE state IN ('MSG1_SENT', 'MSG2_SENT')
  AND updated_at < datetime('now', '-30 minutes')
  AND platform = ?;

-- Напоминание 1 (24 часа, только AWAIT_PAYMENT)
SELECT * FROM users
WHERE state = 'AWAIT_PAYMENT'
  AND updated_at < datetime('now', '-24 hours')
  AND reminder_count = 0
  AND platform = ?;

-- Напоминание 2 (48 часов, только AWAIT_PAYMENT)
SELECT * FROM users
WHERE state = 'AWAIT_PAYMENT'
  AND updated_at < datetime('now', '-48 hours')
  AND reminder_count = 1
  AND platform = ?;
```

## ⚙️ АРХИТЕКТУРА: core/flow.js

`flow.js` — платформонезависимый модуль. Не импортирует ничего из Telegram или VK.

**Принимает:**
```js
handleAction({ platform, chatId, action, text })
```

**Возвращает:**
```js
{
  messages: [
    { text: '...', button: { label: '...', callback: '...' }, delay: 0 },
    { text: '...', delay: 0 },
  ],
  files: ['guide.pdf', 'tracker.pdf', 'wallpaper.jpg'], // или []
  newState: 'MSG2_SENT' // или null если состояние не меняется
}
```

Адаптер получает этот объект и переводит в вызовы конкретного API.

## 🔀 ЛОГИКА ВОРОНКИ (flow.js)

### Действия (action)

| action | Откуда приходит |
|---|---|
| `START` | /start (Telegram) или триггерное слово (VK) |
| `BTN_NEXT_1` | Нажатие кнопки «Читать дальше →» после MSG_1 |
| `BTN_NEXT_2` | Нажатие кнопки «Я готов — к условиям» после MSG_2 |
| `TEXT` | Любое текстовое сообщение |
| `AUTO_PROGRESS` | Вызов из планировщика после 30 минут без действия |

---

### Сценарий 1: Стандартный путь — пользователь нажимает кнопки

```
/start
→ upsertUser state: MSG1_SENT
→ отправить MSG_1 + кнопка [Читать дальше →]
```

```
Нажимает [Читать дальше →]  (BTN_NEXT_1)
→ проверить: state === 'MSG1_SENT'  (защита от повторного нажатия)
→ upsertUser state: MSG2_SENT
→ отправить MSG_2 + кнопка [Я готов — к условиям]
```

```
Нажимает [Я готов — к условиям]  (BTN_NEXT_2)
→ проверить: state === 'MSG2_SENT'
→ upsertUser state: AWAIT_PAYMENT
→ отправить MSG_3
```

```
Пишет ГОТОВО  (TEXT)
→ проверить: state === 'AWAIT_PAYMENT'
→ отправить MSG_4
→ отправить файлы (guide.pdf, tracker.pdf, wallpaper.jpg)
→ upsertUser state: COMPLETED
```

---

### Сценарий 2: Авто-продвижение — не нажал кнопку за 30 минут

Планировщик срабатывает каждые **5 минут** и проверяет пользователей:

```
state === 'MSG1_SENT'  И  updated_at < now - 30 мин
→ upsertUser state: MSG2_SENT
→ отправить MSG_2 + кнопка [Я готов — к условиям]
```

```
state === 'MSG2_SENT'  И  updated_at < now - 30 мин
→ upsertUser state: AWAIT_PAYMENT
→ отправить MSG_3
```

> ⚠️ Когда авто-продвижение отправляет сообщение — `updated_at` обновляется.
> Таймер следующего авто-продвижения отсчитывается заново от этого момента.

---

### Сценарий 3: Пользователь нажимает кнопку после авто-продвижения

```
Состояние уже MSG2_SENT (авто-продвижение сработало)
Пользователь возвращается и нажимает старую кнопку [Читать дальше →]
→ проверить: state !== 'MSG1_SENT'
→ игнорировать нажатие (callback_query.answer без действия)
```

---

### Сценарий 4: Написал что-то вместо ГОТОВО

```
state === 'AWAIT_PAYMENT'  И  текст !== 'ГОТОВО'
→ отправить FALLBACK_PAYMENT[platform]
→ state не меняется
```

---

### Сценарий 5: Повторное взаимодействие после получения файлов

```
state === 'COMPLETED'  +  любое действие (включая /start)
→ отправить FALLBACK_COMPLETED
→ state не меняется
```

---

### Сценарий 6: Пользователь без записи в БД пишет что-то

```
state === null  +  текстовое сообщение (не /start, не триггер VK)
→ отправить FALLBACK_IDLE[platform]
```

---

### Сценарий 7: Напоминания — завис на AWAIT_PAYMENT

```
state === 'AWAIT_PAYMENT'
И updated_at < now - 24 часа
И reminder_count === 0
→ отправить REMINDER_1_AWAIT_PAYMENT
→ incrementReminderCount (count становится 1)
→ state не меняется
```

```
state === 'AWAIT_PAYMENT'
И updated_at < now - 48 часов
И reminder_count === 1
→ отправить REMINDER_2_AWAIT_PAYMENT (финальное)
→ incrementReminderCount (count становится 2)
→ state не меняется
→ больше напоминаний нет
```

> Напоминание для состояний MSG1_SENT / MSG2_SENT не нужно:
> авто-продвижение через 30 минут уже само продвинет пользователя вперёд.

### Таблица всех состояний и переходов

| Текущий state | Действие | Новый state | Ответ бота |
|---|---|---|---|
| null / любой | START | MSG1_SENT | MSG_1 + кнопка |
| MSG1_SENT | BTN_NEXT_1 | MSG2_SENT | MSG_2 + кнопка |
| MSG1_SENT | AUTO_PROGRESS (30 мин) | MSG2_SENT | MSG_2 + кнопка |
| MSG1_SENT | TEXT (любой) | MSG1_SENT | FALLBACK_MID_FUNNEL |
| MSG2_SENT | BTN_NEXT_2 | AWAIT_PAYMENT | MSG_3 |
| MSG2_SENT | AUTO_PROGRESS (30 мин) | AWAIT_PAYMENT | MSG_3 |
| MSG2_SENT | TEXT (любой) | MSG2_SENT | FALLBACK_MID_FUNNEL |
| AWAIT_PAYMENT | TEXT = ГОТОВО | COMPLETED | MSG_4 + файлы |
| AWAIT_PAYMENT | TEXT ≠ ГОТОВО | AWAIT_PAYMENT | FALLBACK_PAYMENT |
| AWAIT_PAYMENT | REMINDER (24ч, count=0) | AWAIT_PAYMENT | REMINDER_1_AWAIT_PAYMENT |
| AWAIT_PAYMENT | REMINDER (48ч, count=1) | AWAIT_PAYMENT | REMINDER_2_AWAIT_PAYMENT |
| COMPLETED | любое (включая /start) | COMPLETED | FALLBACK_COMPLETED |
| MSG1_SENT | BTN_NEXT_1 (повтор) | — | игнорировать |
| MSG2_SENT | BTN_NEXT_1 (устарела) | — | игнорировать |
| AWAIT_PAYMENT | BTN_NEXT_2 (устарела) | — | игнорировать |

## ⏰ ПЛАНИРОВЩИК (scheduler.js)

Один планировщик, два цикла с разным интервалом.

```js
// Авто-продвижение — каждые 5 минут
setInterval(async () => {
  for (const platform of ['telegram', 'vk']) {
    const users = store.getPendingAutoProgress(platform);
    for (const user of users) {
      const result = flow.handleAction({
        platform,
        chatId: user.chat_id,
        action: 'AUTO_PROGRESS',
      });
      await adapters[platform].send(user.chat_id, result);
    }
  }
}, 5 * 60 * 1000);

// Напоминания об оплате — каждые 60 минут
setInterval(async () => {
  for (const platform of ['telegram', 'vk']) {
    const users = store.getPendingReminders(platform); // возвращает тех, кому нужно reminder_count 0→1 или 1→2
    for (const user of users) {
      const text = user.reminder_count === 0
        ? messages.REMINDER_1_AWAIT_PAYMENT
        : messages.REMINDER_2_AWAIT_PAYMENT;
      await adapters[platform].sendText(user.chat_id, text);
      store.incrementReminderCount(platform, user.chat_id);
    }
  }
}, 60 * 60 * 1000);
```
